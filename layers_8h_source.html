19  /* There are two lookup functions, because we have two caches, and different
20  * parts of the firmware will want to use either this or that (or perhaps
21  * both, in rare cases).
22  *
23  * First of all, we use caches because looking up a key through all the layers
24  * is costy, and the cost increases dramatically the more layers we have.
25  *
26  * Then, we have the `liveCompositeKeymap`, because to have layer behaviours
27  * we want, that is, if you hold a key on a layer, release the layer key but
28  * continue holding the other, we want for the layered keycode to continue
29  * repeating.
30  *
31  * At the same time, we want other keys to not be affected by the
32  * now-turned-off layer. So we update the keycode in the cache on-demand, when
33  * the key is pressed. (see the top of `handleKeyswitchEvent`).
34  *
35  * On the other hand, we also have plugins that scan the whole keymap, and do
36  * things based on that information, such as highlighting keys that changed
37  * between layers. These need to be able to look at a state of where the
38  * keymap *should* be, not necessarily where it is. The `liveCompositeKeymap`
39  * is not useful here. So we use `activeLayers` which we update whenever
40  * layers change (see `Layer.on` and `Layer.off`), and it updates the cache to
41  * show how the keymap should look, without the `liveCompositeKeymap`-induced
42  * behaviour.
43  *
44  * Thus, if we are curious about what a given key will do, use `lookup`. If we
45  * are curious what the active layer state describes the key as, use
46  * `lookupOnActiveLayer`.
47  */
48  static Key lookup(byte row, byte col) {
49  return liveCompositeKeymap[row][col];
50  }

52  uint8_t layer = activeLayers[row][col];
